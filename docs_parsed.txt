Sheet: 개요
						
	iFlow Sentinel — 기능요건 정의서					
	SAP Integration Suite 프로젝트 규칙 준수 체커 (Rule Compliance Checker)					
						
	문서명	iFlow Sentinel 기능요건 정의서				
	작성자	박세영				
	작성일	2026-07-14				
	버전	v1.2  (설계서 v1.2 반영: 전역/프로젝트 규칙 모델, 사용자 정의 조건식)				
	통신 방식	SAP Integration Suite OData API 직접 통신 (HTTPS, ICC 미적용)				
	구현 형태	Java(Spring Boot) 백엔드 + Vue 프론트엔드 웹서비스				
						
	■ 범례					
	우선순위	P0 : 필수 핵심 (MVP)  |  P1 : 주요  |  P2 : 향후/부가				
	구분	필수 : 서비스 성립에 반드시 필요  |  선택 : 상황에 따라 선택 적용				
						
	■ v1.2 변경 요약 (설계서 v1.2 대비)					
	•	규칙셋(Ruleset) 생성·상속(import)·프로젝트 바인딩·오버라이드 요건 삭제				
	•	전역 규칙 / 프로젝트 규칙 구분 및 프로젝트별 개별 적용(ProjectRule) 요건 신설				
	•	사용자 정의 조건식(custom-expression) 규칙 — 비주얼 빌더·시험 실행·스키마 탐색기 요건 신설				
	•	테넌트 : 프로젝트 = N : 1 관계 반영 (테넌트는 프로젝트에 종속, 환경 구분 추가)				
	•	이관 관련 리포트 문구를 '차단'에서 '보류 권고'로 조정

1. 문서 개요
1.1 목적
본 문서는 SAP Integration Suite(이하 IS) 통합 프로젝트의 산출물(iFlow 등 아티팩트)이 사전에 정의한 개발 표준·규칙을 준수하는지 자동으로 검사하는 웹서비스 «iFlow Sentinel»의 설계를 기술한다. 개발팀이 별도 배경지식 없이도 시스템의 구조와 동작 원리를 이해할 수 있도록 작성한다.
1.2 대상 독자
본 서비스를 개발·유지보수하는 개발자
표준 규칙을 정의·운영하는 통합 아키텍트 및 리드
도입 여부를 검토하는 관리자
1.3 범위
포함 : 테넌트 연동, 아티팩트 수집·파싱, 전역·프로젝트 규칙 정의·관리(UI/YAML)와 프로젝트별 선택 적용, 사용자 정의 판단 로직, 검사 실행, 결과 리포트.
미포함 : 운영 장애 디버깅(별도 모듈), 배포 자동화, iFlow 편집 기능.
설계 전제 : 본 서비스는 SAP IS 테넌트와 공식 OData API를 통해 HTTPS로 직접 통신한다. 내부 터널링(ICC)은 적용하지 않는다. IS가 공개 HTTPS 엔드포인트를 제공하므로 별도 터널 계층 없이 표준 OAuth2 인증으로 접근 가능하기 때문이다.

2. 시스템 개요
2.1 배경 및 문제 정의
통합 개발 표준은 대개 문서(Word/위키)로만 존재하고, 실제 iFlow가 그 표준을 지키는지는 이관(Transport) 직전에 사람이 눈으로 하나씩 검토한다. 이 방식은 개발자 수와 인터페이스 수에 비례해 비용이 커지고, 검토자의 숙련도에 따라 결과가 달라지며, 누락이 발생한다. 즉 «표준이 실행 가능한 형태로 강제되지 않는» 것이 근본 문제다.
기존 오픈소스 도구(CPILint 등)는 이 문제를 규칙 기반 검사로 풀지만 명령줄(CLI) 전용이고 규칙을 XML로 관리해 진입장벽이 있으며, 웹 UI·팀 단위 거버넌스 워크플로우는 제공하지 않는다. iFlow Sentinel은 이 공백을 «웹서비스 + UI 기반 규칙 정의 + 전역·프로젝트 규칙의 선택 적용»으로 채운다.
2.2 목표 및 핵심 가치
표준을 실행 가능한 규칙으로 코드화하여 사람 대신 기계가 일관되게 검사한다.
규칙 정의를 UI로 쉽게 하되, 저장은 YAML로 하여 이식성·버전관리·재사용을 확보한다.
전역 규칙과 프로젝트 규칙을 나누어, 프로젝트마다 필요한 규칙을 자유롭게 선택·적용한다.
미리 정의된 타입으로 표현할 수 없는 검사도 사용자가 직접 판단 로직을 선언할 수 있게 한다.
검사 결과를 이관 검토 리포트로 제공하여 리뷰 시간을 단축한다.
2.3 주요 특징
특징

3.3 통신 방식
백엔드는 IS 테넌트가 제공하는 공식 OData API에 HTTPS로 직접 접속한다. 인증은 BTP에서 발급한 서비스 키 기반 OAuth2 Client Credentials 방식을 사용한다. 별도 터널 계층(ICC)을 두지 않으므로 구조가 단순하고, 사내 서버 한 대에 백엔드를 배포하면 바로 운영할 수 있다.

적용 방식
전역 규칙 (isGlobal=true)
조직 전체 기준으로 사전에 정의. 여러 프로젝트가 공유.
각 프로젝트가 사용할지 여부를 선택(ProjectRule).
프로젝트 규칙 (customProject)
특정 프로젝트에 한해 정의. 그 프로젝트에만 존재.
해당 프로젝트에서만 사용 여부를 선택.
한 프로젝트에 적용할 규칙 집합은 «사용할 전역 규칙 + 사용할 프로젝트 규칙»으로 구성되며, 각 규칙을 켜고 끄는 것은 프로젝트 내부에서 ProjectRule(project, rule, isEnabled)로 관리한다. 전역 규칙의 값을 프로젝트에서 다르게 쓰고 싶으면, 상속·오버라이드 대신 «전역 규칙을 끄고, 원하는 값으로 프로젝트 규칙을 새로 정의해 켜는» 방식으로 독립적으로 선언한다. 이렇게 하면 전역 규칙 원본은 어떤 경우에도 프로젝트에 의해 변형되지 않는다.
5.5 YAML 예시
전역 규칙과 프로젝트 규칙은 같은 규칙 스키마를 공유하며, isGlobal/customProject로 구분된다. 프로젝트는 별도의 적용 목록으로 어떤 규칙을 켤지 선언한다.
전역 규칙 (라이브러리)
rule:
  ruleKey: sender-naming
  isGlobal: true
  type: naming-convention
  severity: fail
  target: { element: participant, role: sender }
  params:
    prefix: ["OP_", "B2B_", "CP_"]
  message: "송신 시스템 이름은 OP_/B2B_/CP_ 접두사를 사용해야 합니다."
프로젝트 규칙 (사용자 정의 조건식)
rule:
  ruleKey: mapping-step-limit
  isGlobal: false
  customProject: soil-is-migration
  type: custom-expression
  severity: warn
  params:
    schemaVersion: 1
    expression: steps.filter(s, s.type == "mapping").size() <= 5
  message: "매핑 스텝이 5개를 초과했습니다."
프로젝트의 규칙 적용 선택 (ProjectRule)
project: soil-is-migration
appliedRules:
  - { rule: sender-naming,           isEnabled: true }   # 전역 규칙 사용
  - { rule: mapping-step-limit,      isEnabled: true }   # 프로젝트 규칙 사용
  - { rule: allowed-script-language, isEnabled: false }  # 이 프로젝트에선 미적용
5.6 사용자 정의 판단 로직 (Custom Expression)
5.6.1 배경
미리 정의된 규칙 타입은 UI 매핑과 안전성 면에서 유리하지만, 카탈로그에 없는 «새로운 판단 로직»은 표현할 수 없다. 예를 들어 «매핑 스텝이 5개를 넘으면 안 된다»처럼 기존 어떤 타입에도 해당하지 않는 조건은 사용자가 직접 선언할 수 있어야 한다. custom-expression 타입은 이 확장 지점을 제공한다.
5.6.2 접근 방식 — 코드 실행이 아닌 «읽기 전용 질의»
범용 스크립트(Groovy/JS 등)를 그대로 실행하면 사용자가 서버에서 임의 코드를 돌리는 셈이 되어, 리소스 남용이나 타 프로젝트 데이터 접근 등의 위험이 생긴다. 따라서 «코드 실행»이 아니라 «파싱 모델에 대한 부작용 없는 읽기 전용 질의»만 허용한다. 이는 정책 검사 도구(OPA 등)가 범용 언어 대신 조회 특화 선언형 언어를 쓰는 방식과 같다. custom-expression은 기존 «타입 + 파라미터» 구조를 그대로 유지하며, 그 파라미터가 조건식이 될 뿐이다. 따라서 규칙 빌더·YAML 저장·프로젝트 선택 구조가 모두 재사용된다.

5.6.4 백엔드 평가 원리
조건식은 .iflw 원본이 아니라 Parser가 만든 파싱 모델(항상 같은 스키마)을 대상으로 평가된다. 실행 흐름은 다음과 같다.
Connector가 아티팩트를 받아오고, Parser가 파싱 모델을 만든다. (기존 컴포넌트 재사용)
파싱 모델을 표현식 엔진이 읽을 수 있는 «평가 컨텍스트»로 구성한다. (사실상 파싱 모델의 직렬화)
Rule Engine이 CEL 런타임에 조건식과 컨텍스트를 넘겨 평가한다. 조건식은 참/거짓이 아니라 «위반 요소의 목록»을 반환하도록 규정한다.
파싱 모델의 각 노드에는 위치 메타데이터(요소 id·사람이 읽는 위치)가 이미 박혀 있어, 반환된 요소마다 이를 그대로 Finding에 옮긴다.
개수형 조건처럼 요소 목록이 아닌 불리언이 나오는 경우는 «흐름 전체» 스코프의 단일 Finding으로 처리한다.
즉 백엔드가 iFlow의 내용을 «알아서» 해석하는 것이 아니라, Parser가 모든 iFlow를 동일한 스키마로 통일해 두고, 사용자의 조건식은 그 고정 스키마 위에서만 질의하므로 임의의 iFlow에도 재사용된다.
5.6.5 안전장치
시험 실행 : 저장 전에 샘플 iFlow에 대해 미리 실행해 무엇을 잡아내는지 보여준다. 조건을 잘못 짜 «아무것도 안 걸리는»(검사가 조용히 무력화되는) 실패를 방지한다.
평가 시간·복잡도 제한 : 어떤 조건식이라도 검사 1회가 정해진 시간 안에 끝나도록 상한을 둔다.
스키마 탐색기 : 어떤 경로(steps, participants, channels.adapter 등)가 존재하는지 UI에서 탐색·자동완성해, 사용자가 정확한 경로로 조건을 작성하게 한다.
schemaVersion : 규칙에 파싱 모델 스키마 버전을 함께 저장하고, Parser 스키마가 바뀌면 구버전 조건식을 감지해 경고한다. 필드명 변경으로 조건식이 조용히 깨지는 것을 막는다.
6. 처리 흐름
6.1 전체 흐름
사용자 관점의 전체 사용 흐름은 다음 6단계로 요약된다.
① 테넌트 등록 — 검사 대상 IS 테넌트의 OData URL·OAuth2 자격증명을 등록하고 연결을 검증한다.
② 규칙 정의 — UI 규칙 빌더로 전역·프로젝트 규칙을 정의한다. (필요 시 사용자 정의 조건식 포함, YAML 자동 생성)
③ 프로젝트 구성 — 프로젝트를 정의하고, 적용할 전역·프로젝트 규칙을 선택한다(ProjectRule).
④ 검사 실행 — 프로젝트/환경을 선택해 검사를 실행한다. (온디맨드)
⑤ 결과 확인 — pass/warn/fail 요약과 위반 상세를 확인한다.
⑥ 리포트 활용 — 검토 체크리스트/리포트를 내보내 리뷰·티켓에 활용한다.
6.2 검사 실행 시퀀스 (내부)
1. Connector가 대상 아티팩트 목록을 조회하고 각 아티팩트 ZIP을 다운로드한다.
2. Parser가 ZIP을 해제해 .iflw와 리소스를 파싱하여 파싱 모델을 만든다.
3. Rule Engine이 프로젝트에서 활성화된(ProjectRule) 전역·프로젝트 규칙을 로드한다.
4. 파싱 모델을 평가 컨텍스트로 구성한다. (custom-expression 규칙은 이 컨텍스트 위에서 평가된다)
5. 단일 규칙은 아티팩트별로, cross 규칙 및 조건식은 전체 모델을 대상으로 평가한다.
6. 위반은 Finding으로 수집되어 검사 이력에 저장된다.
7. Report가 Finding을 요약·상세 리포트로 가공한다.

9. 보안 설계
자격증명 암호화 : 테넌트 OAuth2 자격증명은 암호화하여 저장하고, 화면·로그에 노출하지 않는다.
전 구간 TLS : 클라이언트↔백엔드, 백엔드↔IS 통신 모두 HTTPS.
인증·인가 : 사용자 로그인 및 역할 기반 권한(RBAC)으로 규칙 변경·검사 실행을 통제.
조건식 격리 : 사용자 정의 조건식은 읽기 전용·부작용 없는 평가로 제한하고, 시간·복잡도 상한을 둔다.
감사 로그 : 규칙 변경, 검사 실행 등 주요 행위를 기록.
민감정보 최소화 : 검사는 설계 정보(구조)만 사용하며 메시지 payload는 다루지 않는다.
10. 배포 아키텍처
백엔드(Spring Boot)와 프론트엔드(Vue)를 사내 서버에 배포한다. ICC 등 별도 네트워크 계층이 없으므로 구성은 단순하다. 백엔드는 인터넷 구간의 IS 테넌트로 아웃바운드 HTTPS만 필요하며, DB는 사내에 둔다. 컨테이너(Docker) 단위 배포를 기본으로 하되, 초기에는 단일 서버 배포로 충분하다.
